# Dataset Documentation

This document describes the datasets and test cases used to exercise Traceable DFIR Investigator, what the agent was tested against, where the data came from, and what the system found.

The project was tested with deterministic demo fixtures and one exploratory case record. The deterministic fixtures are the primary reproducible dataset for judging because they can be regenerated from source code and produce traceable evidence, trace, and finding records.

## Dataset Summary

| Case ID | Scenario | Source | Purpose | Result |
| --- | --- | --- | --- | --- |
| `CASE-DEMO-PERSISTENCE-001` | Windows persistence / suspicious execution | Built-in fixture defined in `dfir_backend/tools/demo_case.py` | Validate agent workflow, evidence linking, trace linking, report generation, and self-correction | 3 findings, all linked to evidence and traces |
| `CASE-DEMO-EXFIL-001` | Archive staging and possible exfiltration | Built-in fixture defined in `dfir_backend/tools/demo_case.py` | Validate multi-case behavior and exfiltration-style findings | 3 findings, all linked to evidence and traces |
| `CASE-001` | Earlier simulated / exploratory forensic case | `evidence/case-001/simulated_evidence.json` plus later E01-oriented scan records | Validate timeline parsing, process/timeline storage, filesystem scanning, and report iteration during development | Multiple repeated development runs; latest traceable scan recorded 67 Windows Temp entries and 0 suspicious entries |

## Primary Reproducible Dataset

The primary dataset is generated by:

```text
dfir_backend/tools/demo_case.py
```

The demo bundle creates two cases:

```text
CASE-DEMO-PERSISTENCE-001
CASE-DEMO-EXFIL-001
```

The cases can be created from the UI with `LOAD DEMOS`, or through the API:

```bash
curl -X POST http://localhost:8000/api/cases/demo-bundle \
  -H "Content-Type: application/json" \
  -d "{\"reset\": true}"
```

The agent can then be run for each case:

```bash
curl -X POST http://localhost:8000/api/cases/CASE-DEMO-PERSISTENCE-001/agent-run
curl -X POST http://localhost:8000/api/cases/CASE-DEMO-EXFIL-001/agent-run
```

Generated records are written under:

```text
evidence/<CASE_ID>/evidence_store.jsonl
evidence/<CASE_ID>/trace_store.jsonl
evidence/<CASE_ID>/findings_store.jsonl
```

## What The Agent Was Tested Against

The agent was tested against:

- Deterministic fixture records for persistence and exfiltration scenarios.
- Stored evidence records with explicit artifact paths.
- Stored trace records with tool names, worker containers, command arguments, artifact paths, and UI links.
- Stored finding records with evidence references, trace references, rule IDs, severity, confidence, and artifact locations.
- A simulated timeline JSON with Windows event, registry, prefetch, and file metadata examples.
- An E01-oriented filesystem scan path for `CASE-001`.
- Self-correction paths where an intentionally incomplete draft report is rejected and rebuilt from traceable records.

## What The Agent Found

Across the two primary demo cases, the agent found six trace-linked findings:

- 2 critical findings:
  - Suspicious service install pointing to a Temp executable.
  - Possible upload activity after archive creation.
- 3 high findings:
  - Executable found in Windows Temp.
  - Suspicious encoded PowerShell execution.
  - Sensitive archive staged in user Temp.
- 1 medium finding:
  - PowerShell archive creation.

The older `CASE-001` development case additionally exercised findings for Temp execution, lateral movement indicator strings, failed logons, registry persistence, scheduled task creation, service creation, and filesystem scanning.

## Reproducibility Notes

To reproduce the primary dataset:

1. Start the platform.
2. Create the demo bundle with `reset: true`.
3. Run `agent_run_case` for each demo case.
4. Inspect the generated JSONL stores under `evidence/<CASE_ID>/`.
5. Verify that every final finding has at least one `evidence_ref`, one `trace_ref`, and one artifact location.

Expected primary-case counts:

| Case | Evidence Records | Base Tool Traces | Findings |

| `CASE-DEMO-PERSISTENCE-001` | 3 | 3 | 3 |

## Evidence Integrity Approach

The project uses architectural restrictions rather than prompt-only restrictions for evidence integrity.

Key controls:

- The LLM is not the evidence collector or decision-maker. It receives a compact case brief after deterministic tools have already created evidence, traces, and findings.
- Findings are generated from structured tool outputs and stored records, not from free-form model claims.
- Every accepted finding must link back to evidence records and trace records.
- The final report validation rejects findings that lack traceability.
- Worker execution is isolated per case using case-specific worker containers.
- Source evidence paths are represented separately from generated stores. Evidence, trace, and finding records are written as derived metadata under the case storage path.
- The UI and LLM report layers treat LLM text as commentary; `llm_explanation_is_evidence` is false in stored findings.

If a model ignores a prompt restriction, it still cannot directly create validated evidence because the architecture requires durable evidence and trace IDs generated by deterministic code. A hallucinated claim may appear only as commentary unless it is backed by stored evidence and trace references.

## Spoliation Testing Notes

The demo fixture path tests spoliation resistance by design: fixtures are source definitions, while generated evidence, trace, and finding records are derived outputs. Running the agent writes JSONL records and orchestration traces, but it does not need to modify the original fixture definitions.

For real E01-style workflows, the architecture is intended to mount evidence into worker containers and run forensic tools against mounted paths. The latest `CASE-001` filesystem trace shows tool execution against:

```text
/mnt/ewf/CASE-001/ewf1
```

Known limitation:

- The current stored trace metadata has `source_sha256: null` for the demo fixtures and `CASE-001` scan. For stronger formal evidence integrity reporting, future runs should record a SHA-256 hash for each original evidence file before and after analysis and include explicit read-only mount flags in the trace metadata.

This is a documented limitation, not a hidden assumption. The system already separates original evidence paths from derived records, but formal before/after hash verification would strengthen spoliation testing for real evidence images.

## Limitations

- The primary demo cases are deterministic synthetic fixtures, not third-party forensic corpora.
- `CASE-001` includes repeated development output and should not be treated as a clean benchmark dataset.
- Demo fixture trace records currently do not include source SHA-256 hashes.
- The Railway preview deployment can show the API/frontend and deterministic demo behavior, but full SIFT worker orchestration depends on Docker daemon access and is most reproducible with Docker Compose locally or on a controlled Linux host.
